## 3. 技术风险评估

| 风险 | 等级 | 影响范围 | 缓解措施 |
|------|------|----------|----------|
| SQLite 单文件数据库在并发写入时容易成为瓶颈 | 高 | 客户更新、跟进记录、新建排期、提醒完成等写操作高峰 | 启用 WAL 和 busy timeout，减少长事务；把写操作控制得短小明确；后续数据量上来后评估迁移 PostgreSQL。 |
| 无 Composer、无框架导致基础设施代码完全自维护 | 中 | 路由、验证、错误处理、模板渲染、迁移脚本、备份脚本 | 统一约定目录与命名；把公共能力沉到 `app/Core` 和 `app/Support`；补齐最少量单元测试和冒烟脚本。 |
| 发帖间距/提醒策略偏简单，难覆盖真实运营节奏 | 中 | 营销排期、断更提醒、冲突判断 | 先用可配置的 `posting_rules` 表承载规则；提醒逻辑集中在服务层；给后续按渠道/团队细分留扩展点。 |

## 4. 技术可行性分析

### 4.1 核心功能可行性

| 功能 | 可行性 | 复杂度 | 说明 |
|------|--------|--------|------|
| 客户与潜客资料管理 | ✅ 可行 | M | 典型 CRUD + 搜索筛选，SQLite 和原生 PHP 足以承载，关键是把字段约束和列表体验做稳。 |
| 跟进记录与回访提醒 | ✅ 可行 | M | 跟进历史采用独立表最稳妥，提醒本质是按时间计算待办，适合放在服务层统一处理。 |
| 营销帖排期与发帖间距提醒 | ✅ 可行 | M | 排期与规则解耦后实现难度可控，真正需要注意的是规则表达要够清晰，避免过早写死。 |

### 4.2 技术难点

| 难点 | 解决方案 | 预估工时 |
|------|----------|----------|
| SQLite 并发与事务边界控制 | 所有写操作使用短事务，尽量避免跨多个表的长时间锁定；初始化时统一开启外键和 WAL。 | 0.5-1 天 |
| 手写路由、校验、CSRF 和模板复用 | 建立统一的 `Core` / `Support` 基础层，所有控制器和表单提交都走同一套请求与响应封装。 | 1-2 天 |
| 发帖间距规则与提醒生成 | 将规则读写集中在 `PostingRuleService` / `ReminderService`，并先支持“最小间隔”和“最大断更”两种阈值。 | 1 天 |

## 7. 代码规范建议

### 7.1 目录结构规范

```
sales-crm/
├── public/                 # 仅暴露 Web 入口和静态资源
├── app/
│   ├── Core/               # Request/Response/Router/DB/CSRF 等基础能力
│   ├── Controllers/        # 控制器只做请求编排，不拼业务 SQL
│   ├── Domain/
│   │   ├── Repositories/   # 数据访问
│   │   └── Services/       # 业务规则与流程
│   ├── Support/            # 校验、格式化、辅助函数
│   └── Views/              # 模板只负责展示
├── config/                 # 配置文件
├── database/               # schema、seed、sqlite 文件
├── scripts/                # 初始化、备份、维护脚本
├── storage/                # 日志和备份产物
└── tests/                  # 单元测试与功能测试
```

### 7.2 命名规范

- **文件命名**：控制器、服务、仓储使用 `PascalCase.php`，脚本使用 `kebab-case` 或 `snake_case` 保持一致，但同一目录内不要混用多种风格。
- **组件命名**：类名使用明确职责命名，例如 `ContactController`、`ReminderService`、`MarketingPostRepository`。
- **函数命名**：控制器动词开头，服务层表达业务意图，例如 `store()`, `updateReminder()`, `calculatePostingGap()`。
- **变量命名**：PHP 变量统一 `camelCase`，布尔值用 `is/has/can` 前缀，避免缩写过度。

### 7.3 代码风格

- 控制器薄、服务厚、仓储单一职责，SQL 只允许出现在仓储层。
- 每个写接口都必须做服务端校验、CSRF 校验和统一错误返回，不依赖前端兜底。
