- **核心目标**：帮助销售团队在一个系统里统一管理客户、潜在客户、跟进记录和营销帖排期，减少表格分散、遗漏跟进和发帖节奏失控的问题。
- **目标用户**：销售团队成员为主，管理层为次要查看方。
- **关键功能**：客户与潜客资料管理、跟进记录与回访提醒、潜客阶段推进、营销帖排期管理、发帖间距提醒、KPI 看板、基础角色权限。
- **技术约束**：技术偏好为 `PHP`；当前版本保持零依赖单体实现，只做 `manager / sales` 两级权限，不引入复杂企业级 RBAC 或重型 BI。
- **优先级**：首版闭环已交付，当前迭代补齐 KPI 看板与基础角色权限，复杂组织架构、自动化营销和多维 BI 仍放在后续版本。

## 1. 概述

### 1.1 背景
销售团队当前需要同时处理老客户维护、潜在客户推进和营销内容发布，但信息通常分散在表格、聊天记录和个人记事中，容易出现跟进遗漏、客户状态不一致、发帖过密或断更等问题。首版产品需要先把最核心的日常执行动作收拢到一个系统里，形成可持续使用的业务闭环。

### 1.2 目标
建立一个面向销售团队的轻量 CRM，支持客户与潜客信息的统一管理、跟进和回访记录的持续沉淀、营销帖排期与发帖间距提醒，并通过 KPI 看板和基础角色权限让销售与管理层看到各自需要的业务视图。

### 1.3 成功指标
- [ ] 销售可以在系统内完成客户/潜客建档、跟进记录和回访提醒配置，日常使用不再依赖外部表格。
- [ ] 营销帖排期可以被清晰查看，系统能够在发帖间隔过短或过长时给出提醒，减少人为疏漏。

## 3. 功能需求

### FR-001：客户与潜客资料管理
- **描述**：当销售需要新增、查看或更新客户/潜客信息时，系统应支持维护基础档案，包括名称、联系方式、来源、阶段状态、负责人、备注等信息，并区分现有客户与潜在客户。
- **优先级**：必须有
- **依赖**：基础数据存储
- **验收标准**：
  - [ ] AC-1：销售可以创建、编辑、查看和检索客户/潜客档案。
  - [ ] AC-2：系统能够区分“现有客户”和“潜在客户”，并展示其当前状态。

### FR-002：跟进记录与沟通历史
- **描述**：当销售完成一次沟通、拜访或线上联系时，系统应允许记录跟进内容、跟进时间、结果和下一步动作，并保留历史记录供后续查看。
- **优先级**：必须有
- **依赖**：客户与潜客资料管理
- **验收标准**：
  - [ ] AC-1：每个客户/潜客都可以关联多条跟进记录。
  - [ ] AC-2：历史记录按时间倒序展示，能够回看完整沟通链路。

### FR-003：回访提醒与待办管理
- **描述**：当销售为客户或潜客设置回访时间时，系统应在到期前或到期时发出提醒，并在列表中形成待办，避免重要客户被遗漏。
- **优先级**：必须有
- **依赖**：跟进记录与沟通历史
- **验收标准**：
  - [ ] AC-1：销售可以为单条跟进记录或客户档案设置下次回访时间。
  - [ ] AC-2：到期的回访任务能在待办列表或提醒区中被查看到。

### FR-004：潜客阶段推进
- **描述**：当销售推进潜客进入下一业务阶段时，系统应支持修改阶段状态，例如新线索、已联系、已跟进、待成交等，以便体现当前推进进度。
- **优先级**：必须有
- **依赖**：客户与潜客资料管理
- **验收标准**：
  - [ ] AC-1：系统支持对潜客阶段进行更新。
  - [ ] AC-2：列表页可以直接看到当前阶段，便于快速筛选和跟进。

### FR-005：营销帖排期管理
- **描述**：当运营或销售需要规划营销帖子发布时，系统应支持记录帖子标题、计划发布时间、内容摘要、负责人和发布状态，形成可查看的排期列表。
- **优先级**：必须有
- **依赖**：无
- **验收标准**：
  - [ ] AC-1：可以创建、编辑和查看帖子排期。
  - [ ] AC-2：排期列表可以按时间顺序展示未来和已发布内容。

### FR-006：发帖间距提醒
- **描述**：当营销帖发布时间过于密集或距离上次发布时间过久时，系统应基于预设间隔规则给出提醒，帮助团队保持稳定发帖节奏。
- **优先级**：必须有
- **依赖**：营销帖排期管理
- **验收标准**：
  - [ ] AC-1：系统能够在创建或调整排期时检测发帖间隔是否异常。
  - [ ] AC-2：系统能提示“过密”或“断更”风险，并显示最近一次发帖时间。

### FR-007：基础搜索与筛选
- **描述**：当销售需要快速找到目标客户、潜客或帖子时，系统应支持按名称、阶段、负责人、时间范围等条件进行搜索和筛选。
- **优先级**：应该有
- **依赖**：客户与潜客资料管理、营销帖排期管理
- **验收标准**：
  - [ ] AC-1：列表页支持至少按关键字和状态筛选。
  - [ ] AC-2：筛选结果与查询条件一致，可用于日常检索。

### FR-008：基础角色权限
- **描述**：当不同角色登录系统时，系统应根据 `manager` 和 `sales` 两类角色展示对应的数据范围。管理层可查看团队全量数据，销售仅查看自己负责的客户、提醒和营销排期。
- **优先级**：第二阶段必须有
- **依赖**：认证、客户与营销数据模型
- **验收标准**：
  - [ ] AC-1：`sales` 账号无法在列表和详情中看到他人负责的客户、提醒和排期。
  - [ ] AC-2：`manager` 账号可以查看团队全量数据。

### FR-009：KPI 统计看板
- **描述**：当销售或管理层进入 KPI 页时，系统应按角色展示近 30 天客户池结构、跟进执行、提醒压力和营销排期完成情况，帮助判断业务节奏是否健康。
- **优先级**：第二阶段必须有
- **依赖**：客户、提醒、跟进、营销排期
- **验收标准**：
  - [ ] AC-1：销售可查看自己的 KPI 快照和个人结构。
  - [ ] AC-2：管理层可查看团队 KPI、成员对比和业务压力概览。

## 4. 非功能需求

### NFR-001：性能要求
- **类型**：性能
- **描述**：首版系统应支持销售日常高频查看和录入，常用列表和详情页应保持快速响应，避免影响一线使用体验。
- **指标**：常规列表与详情页在正常数据量下应在 2 秒内完成主要内容加载。

### NFR-002：安全要求
- **类型**：安全
- **描述**：系统应避免客户信息、跟进记录和营销排期被未授权访问或误删，基础操作需有明确的访问控制和审计痕迹。
- **指标**：关键写操作需记录操作人和时间；敏感数据传输应使用安全连接。

### NFR-003：可用性要求
- **类型**：可用性
- **描述**：首版界面应尽量减少学习成本，核心流程需要能在少量步骤内完成，适合销售高频、碎片化使用。
- **指标**：核心任务路径尽量控制在 3 步以内完成主要操作。

## 5. 用户故事

### US-001：记录潜客跟进
- **作为** 销售专员
- **我想要** 快速创建潜客并记录每次跟进情况
- **以便** 我能持续推进成交，不遗漏任何沟通节点
- **验收标准**：
  - [ ] 可以新增潜客并补充基础信息。
  - [ ] 每次跟进后都能留下时间、内容和下一步动作。

### US-002：管理老客户回访
- **作为** 销售专员
- **我想要** 为老客户设置回访提醒
- **以便** 我不会忘记重要客户的维护节奏
- **验收标准**：
  - [ ] 可以设置下次回访时间。
  - [ ] 到期后能在待办中看到提醒。

### US-003：规划营销帖发布
- **作为** 营销或销售成员
- **我想要** 在系统里安排营销帖发布时间
- **以便** 我能保持稳定发帖节奏，避免过密或断更
- **验收标准**：
  - [ ] 可以创建帖子排期并查看列表。
  - [ ] 系统能提示发帖间距异常。

## 6. 范围定义

### 6.1 范围内
- 客户与潜客资料管理
- 跟进记录、沟通历史、回访提醒
- 潜客阶段推进
- 营销帖排期管理
- 发帖间距提醒与断更提醒
- 基础搜索与筛选
- 基础角色权限（`manager / sales`）
- KPI 统计看板（近 30 天执行视图）

### 6.2 范围外
- 组织架构级的细粒度权限、审批流和多层级数据隔离
- 复杂自动化营销编排和外部系统深度集成
- 复杂 BI 分析、多时间窗口对比和自定义图表搭建

- [ ] 回访提醒在首版中是仅展示待办，还是需要支持站内通知/邮件提醒。
